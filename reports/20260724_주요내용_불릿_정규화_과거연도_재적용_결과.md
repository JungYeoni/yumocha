# [전처리] 주요내용 불릿 정규화 과거 연도 재적용 결과 — 2026-07-24

- 이슈: #34
- 대상: `data/interim/{시도}/{연도}_{시도}_세부사업_정제.csv`, `..._정제_long.csv`의 `주요내용`, `주요내용_정제` 컬럼

## 작업 배경

`dedup_label()`(공통 유틸)이 개선된 이후에도, 이미 저장된 과거 연도 산출물에는 개선 전 결과가 남아 있었다. LLM 재호출은 필요 없다고 판단해(체크포인트 파일 보존 확인) 규칙 기반 재적용만 수행했다.

## 잔존 현황 재확인

이슈 작성 시점에는 2016·2021은 해당 사례가 없다고 기록됐으나, 재확인 결과 두 연도 모두 잔존이 발견됐다. 2016 노트북을 여러 차례 재실행하면서 산출물이 다시 생성됐고, 그 과정에서 `주요내용_정제`(LLM 체크포인트 결과)에 LLM이 문장을 분리하며 자체적으로 삽입한 불릿(`•` 등)이 다시 남은 것으로 추정된다.

| 연도 | `주요내용` 잔존 | `주요내용_정제` 잔존 |
|---|---:|---:|
| 2016 | 0 | 650 |
| 2017 | 697 | 686 |
| 2021 | 114 | 110 |
| 2022 | 2,046 | 2,046 |
| 2023 | 21 | 7 |
| 2024 | 15 | 5 |

## 처리 방법

1. `BULLET_PATTERN`(`ㅇ○◦□▪•` 및 한글·영숫자 뒤가 아닌 `·`)으로 잔존 행을 먼저 식별
2. 재적용 전, 잔존 행 전체에 대해 `numbers_preserved()`로 숫자 시퀀스·순서가 보존되는지 검증 — **전 연도·전 컬럼 숫자 불일치 0건**
3. 잔존이 있는 파일만 `dedup_label()` 재적용 후 `utf-8-sig`로 재저장(wide·long 동일하게 처리)
4. 재적용 후 전체 파일에서 잔존 재검사 — **최종 잔존 0건**

재적용 과정은 `scripts/reapply_bullet_normalization.py`로 재현할 수 있다. 기본 실행은
파일을 변경하지 않는 검증 모드이며, `--write`를 지정하면 변경 전 파일을
`data/interim/.bullet_normalization_backups/{실행시각}/`에 백업한 뒤 원본을
원자적으로 교체한다.

```bash
# 잔존·숫자/수량 보존·wide/long 동기화 검증만 수행
uv run python scripts/reapply_bullet_normalization.py

# 백업 생성 후 실제 CSV에 반영
uv run python scripts/reapply_bullet_normalization.py --write
```

## 참고

- `data/interim/`은 `.gitignore` 대상이라 이번 CSV 수정 자체는 git diff에 나타나지 않는다. 이 보고서가 작업 근거·검증 기록이다.
- macOS에서 일부 한글 파일명이 분해형(NFD)으로 저장된 경우도 누락하지 않도록 파일 탐색과 wide·long pair 검증 시 NFC 정규화를 적용한다.
- `run_checkpointed_refinement()`는 신규 LLM 응답뿐 아니라 재사용하는 기존 체크포인트의 `주요내용_정제`에도 `dedup_label()`을 적용한다. 정규화로 값이 바뀌고 신규 호출 대상이 없으면 체크포인트 자체도 즉시 다시 저장하므로, 노트북 재실행 시 과거 불릿이 산출물로 되돌아오는 문제를 방지한다.
