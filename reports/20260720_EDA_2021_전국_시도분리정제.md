## 1. 작업 목적

2024년도 전국 시도 분리·정제 파이프라인을 기준으로 2021년도 지방자치단체 저출산·고령사회 시행계획의 세부사업 예산 데이터를 정제하였다.

2021년도 자료의 지역별 세부사업, 사업분류, 당해예산·전년도예산, 주요내용을 분석 가능한 구조로 변환하고, 소계 대조 및 원문 검증을 통해 데이터 품질을 확인하였다.

## 2. 입력 데이터

- 파일: `세부사업표추출_2021년도 지방자치단체 저출산고령사회 시행계획 (제4차 기본계획)_칼럼정렬.xlsx`
- 입력 시트: `정리본_자동`
- 입력 규모: 8,115행 × 12열
- 대상 지역: 전국 17개 시도
- 당해예산: 2021년 예산
- 전년도예산: 2020년 예산

입력 파일은 `data/raw/칼럼정렬/`에서 읽었으며 원본 파일은 수정하지 않았다.

## 3. 처리 방법

1. `정리본_자동` 시트 로드
2. 지역별 데이터 분리
3. 대분류 소계·중분류 소계·세부사업·반복 헤더 분류
4. 대분류·중분류 라벨을 세부사업 행에 전파
5. 실제 세부사업 7,301행 추출
6. 당해예산·전년도예산 숫자형 변환
7. 증감액·증감율 직접 재계산
8. 주요내용 텍스트 정규화
9. 정규식 기반 지원대상·지원내용 부분 추출
10. Solar Pro 3 기반 주요내용 보존형 교정
11. 숫자 보존 검증 후 `주요내용`·`주요내용_정제` 문장 앞 불릿 후처리
12. wide·long 데이터 생성
13. 중분류 소계 QA 및 최종 산출물 검증

## 5. 중분류 소계 QA

중분류 소계 133건을 대상으로 원본 소계와 세부사업 합계를 비교하였다.

- 정확히 일치: 113건
- 미세차이: 18건
- 원본 소계 불일치: 2건

2차 멘토링의 5~10% 허용범위 제안에 따라 절대 오차율 10%를 기본 QA 기준으로 추가했다. 재계산 결과 133건 모두 10% 이내였으며 최대 절대 오차율은 4.98%였다.

- 10% 허용범위 이내: 133건
- 10% 초과: 0건
- 판정불가: 0건

미세차이는 대부분 0.04–70.9 수준으로 반올림 또는 원본 집계 단위 차이로 판단하였다.

별도 확인 대상으로 분류한 2건은 다음과 같다.

- 경북 자체사업 `2. 건강하고 능동적인 고령사회 구축`: 차이 -3,569.6
- 충북 자체사업 `3. 모두의 역량이 고루 발휘되는 사회`: 차이 -462.02

두 건은 세부사업 행 누락보다는 원본 문서의 소계와 기재된 세부사업 합계가 일치하지 않는 사례로 판단하였으며, 값을 임의로 보정하지 않았다.

QA 전체 결과는 `reports/2021_전국_QA_검증결과.csv`에 저장하였다.

## 6. 주요내용 LLM 정제

공개 행정자료의 주요내용 7,301건에 대해 Solar Pro 3를 이용한 보존형 교정을 수행하였다.

LLM은 다음 작업만 수행하도록 제한하였다.

- 잘못 삽입된 공백 정리
- 줄바꿈으로 끊긴 단어 정리
- 명백한 OCR 문자 오류 교정
- 중복 라벨 정리

숫자·금액·비율·날짜·기관명·사업명은 변경하지 않도록 하였으며, 정제 전후 숫자 시퀀스를 비교하였다.

초기 실행에서 숫자 또는 원문이 변경된 3건을 발견하였다.

- 대구 `선천성 대사이상 및 환아관리비 지원`
- 경북 `난청조기진단`
- 제주 `북스타트(Book-start)운동`

세 건은 모두 원문으로 복구하고 체크포인트에도 반영하였다.

- 최종 숫자보존 불일치: 0건
- 검토 기록: `reports/2021_LLM_숫자불일치_검토.csv`

### 6.1 LLM 실행 기록 확인

노트북의 API 설정 셀은 `../.env`에서 `UPSTAGE_API_KEY`를 읽고, 키가 없으면 즉시 `RuntimeError`를 발생시킨다. 해당 셀이 오류 없이 실행된 후 후속 셀까지 순차적으로 실행된 기록이 있으므로, 작업 당시에는 API 키가 설정되어 있었다.

다만 현재 남은 전체 처리 셀의 출력은 다음과 같다.

```text
체크포인트 발견: 7301건 이미 처리됨, 이어서 진행
0it [00:00, ?it/s]
숫자 불일치(검토 대상) 건수: 0
```

즉, 마지막 실행은 7,301건을 신규 API 호출한 것이 아니라 기존 체크포인트를 로드해 병합·검증한 재실행이다. `data/interim/`은 Git 추적 제외 대상이므로, 현재 체크아웃에서 체크포인트 파일이 없는 것은 정상이며 해당 파일은 최초 실행자 환경에서 관리한다.

### 6.2 문장 앞 불릿 후처리

원문과 LLM 정제문장 앞에 `•`, `ㅇ`, `○`, `◦`, `□`, `▪`, `·`, `o`, `-` 등의 불릿이 남아 있는 사례가 확인되었다. 이는 LLM을 다시 호출하지 않고, 체크포인트 병합과 숫자 보존 검증이 끝난 뒤 최종 저장 직전에 결정론적 후처리로 제거하도록 수정했다.

- 적용 컬럼: `주요내용`, `주요내용_정제`
- 제거 범위: 문장 맨 앞의 불릿만 제거
- 보존 범위: 문장 중간의 기호와 결측치는 그대로 유지
- 체크포인트 영향: 없음(후처리 결과를 체크포인트에 다시 저장하지 않음)

### 6.3 코드 재사용성 정비

노트북 상단에 `YEAR = 2021`, `PREVIOUS_YEAR = YEAR - 1`을 선언했다. 예산 컬럼명, 원본 파일명, QA 보고서, LLM 체크포인트, wide·long 산출물 및 최종 검증 glob은 `YEAR`·`PREVIOUS_YEAR`를 사용하도록 변경했다. 추가로 현재 `정리본_자동` 시트가 이미 12개 표준 컬럼만 포함하므로, 범위를 벗어나는 `usecols="C:N"` 지정을 제거했다.

## 9. 한계 및 후속 작업

- 원본 시행계획 문서의 소계와 세부사업 합계가 일치하지 않는 사례가 존재한다.
- 주요내용의 형식이 지역·사업별로 달라 지원대상·지원내용은 정규식이 확실히 적용되는 행만 부분 추출하였다.
- LLM 정제 결과는 숫자보존 검사를 통과했지만, 자연어 전체 의미의 완전한 보존을 보장하지는 않는다.
- 이후 연도 간 사업 비교 시 동일 사업명이 여러 행 존재하는 다대다 병합을 방지하기 위해 사업명 단위 선집계와 `validate="one_to_one"` 검증이 필요하다.
- 후속 정비로 추가한 불릿 제거는 코드에만 반영된 상태이다. `2021_llm_정제_체크포인트.csv`를 보유한 최초 실행자가 체크포인트 병합 → 숫자 보존 검증 → 불릿 후처리 → wide·long 저장·검증 셀을 재실행해 시도별 CSV를 다시 저장해야 한다. 이때 신규 LLM 호출 대상이 `0it`인지 확인하고, 체크포인트는 삭제하지 않는다.
