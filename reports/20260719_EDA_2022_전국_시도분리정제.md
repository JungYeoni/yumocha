# 2022년 전국 시도별 세부사업 예산 현황 분리·정제 결과

## 1. 작업 개요

- 입력 범위: 2022년 전국 17개 시도
- 입력 자료: `data/raw/칼럼정렬/`의 2022년도 칼럼 정렬 원본 Excel, `정리본_자동` 및 `Table 1`
- 실행 노트북: `notebooks/20260719_EDA_2022_전국_시도분리정제.ipynb`
- 작업 목적: 반복 머리글과 소계행을 구분하고 세부사업 leaf를 확정한 뒤, 예산 QA·사업명 중복 후보·주요내용 보존형 정제를 검증하여 시도별 wide·long CSV를 생성하는 것

최종 결과는 leaf 8,122행, long 16,244행이다. 최종 체크포인트와 시도별 산출물은 `data/interim/`에 저장되며 Git ignore 대상이다.

- 후보 원본행: 서울 28
- 병합 대상: 서울 24

- 병합 후 세부사업명: `학대피해아동 보호를 위한 안전망 구축`

## 3. 숫자 변환

예산 숫자 변환에는 노트북의 공통 함수 `to_numeric_budget`를 사용했다.

- 쉼표와 앞뒤 공백 제거
- `"-"`를 0으로 변환
- `"비예산"`을 0으로 변환
- 변환 불가능한 값은 결측으로 처리

당해예산, 전년도예산, 원본 증감액에 같은 변환 함수를 재사용했다. 최종 증감액과 증감율은 변환된 당해·전년도 예산으로 계산했다.

반올림 후 불일치 15건은 다음처럼 분류했다.

## 6. 주요내용 보존형 정제와 전수검토

`주요내용`이 결측이 아닌 LLM 대상은 8,113건이고, 원본 결측은 9건이다. 전체 LLM 실행은 로컬에서 완료했으며, 최종 체크포인트는 구조 이상 7행을 제거한 전체 leaf 인덱스 8,122건을 보유한다.

- LLM 대상: 8,113건
- 원본 `주요내용` 결측: 9건
- 저장된 실행 출력에서 확인된 API 기술 실패 및 최종 원문 유지 실패 메시지: 0건
- 체크포인트 인덱스 중복: 0건

원문과 정제문장이 달라진 319건은 원문·정제문장·지역·원본행·세부사업명을 기준으로 전수검토했다.

| 판정 | 건수 |
|---|---:|
| 공백 정리 | 154 |
| 명백한 오탈자 교정 | 126 |
| 보존 원칙에 따른 원문 유지 | 39 |
| 합계 | 319 |

39건은 **“LLM 오류”가 아니라 “보존 원칙에 따라 원문 유지로 판정한 행”**이다. 내용 삭제·추가, 요약, 사업명 또는 응답 형식 혼입, 의미가 달라질 수 있는 치환, 안전 여부가 애매한 변경을 보수적으로 원문 유지 처리했다.

검토표는 다음 경로에 UTF-8-SIG CSV로 저장했다.

- `reports/2022_LLM_주요내용_정제_319건_전수검토.csv`

노트북의 최종 저장 셀은 이 검토표를 읽어 판정 집계와 39개 키의 중복·누락을 검증한 뒤, `(지역, 원본행, 세부사업명)`으로 정확히 매칭된 39건에 대해 `주요내용_정제 = 주요내용`을 멱등 적용한다. 같은 값을 체크포인트에도 반영한 후 숫자보존을 다시 계산한다.

셀 48의 기존 출력은 39건의 원문 유지 판정을 적용하기 전 숫자보존 불일치를 표시한 오래된 결과여서 제거했다. LLM 완료 상태를 보류로 되돌린 것은 아니며, 최종 결과는 셀 49의 체크포인트 8,122행, 319건 전수검토, 원문 유지 39건 및 강화된 숫자보존 일치 8,122건·불일치 0건 출력으로 검증한다.

숫자보존 검사는 단순 숫자열 비교가 아니라 부호, 소수 및 퍼센트 여부를 포함한 숫자 시퀀스를 비교한다. 유니코드 마이너스는 일반 마이너스로 통일하고, `20-50`과 같은 범위 하이픈은 두 번째 숫자의 음수 부호로 해석하지 않는다. 쉼표는 `1,234.5`처럼 표준 천 단위 형식일 때만 제거하여 숫자값을 정규화하며, `3,00` 같은 비표준 쉼표 표기는 그대로 구분한다. 원문과 정제문장이 모두 결측이면 일치, 한쪽만 결측이면 불일치로 판정한다.

최종 숫자보존 결과:

- 일치: 8,122건
- 불일치: 0건
- 원본 `주요내용` 결측: 9건
- `주요내용_정제` 결측: 9건
- 결측 위치: 동일

## 8. 최종 체크포인트 공유와 무결성

- 로컬 경로: `data/interim/2022_llm_정제_체크포인트.csv`
- 파일명: `2022_llm_정제_체크포인트.csv`
- 행 수: 8,122
- SHA-256: `995d5595face3dfea363907e626364331eef188b794efe3f60448f156977dd4f`
- 저장소 정책: `data/interim/`은 Git ignore 대상이므로 체크포인트와 최종 산출물을 Git에 커밋하지 않음
- Drive 공유 방식: 팀 구성원으로 접근을 제한한 공유 Drive에 공유 완료. 내려받은 파일의 SHA-256을 위 값과 대조
- Drive 위치: 팀 내부 공유 경로와 파일명으로 전달하며, 접근 제한 링크는 공개 저장소에 기록하지 않음

최종 체크포인트는 Git ignore 대상으로 유지하고, SHA-256으로 무결성을 관리한다.

## 9. 한계와 후속 작업

1. 실질 QA 3건은 원본 소계와 leaf 합계의 원천 불일치다. 원문을 임의 보정하지 않았으므로 후속 확인 시 원 작성기관 자료 또는 별도 원천 문서와 대조해야 한다.
2. 미세 차이 12건은 소수점 첫째 자리로 고정한 차이를 별도 분류한 것이다. 분석 목적에 따라 허용 범위를 다시 정의할 수 있으나 현재 CSV 값은 유지했다.
3. near duplicate 288건과 고신뢰 2건은 검토 후보이며 자동 병합·삭제하지 않았다. 사업 담당자의 의미 기반 확인이 필요하다.
4. 보존 원칙에 따라 원문 유지한 39건은 이후 정제 규칙이나 모델을 변경하더라도 동일 검토표를 적용해 재현해야 한다.
5. 최종 체크포인트와 시도별 산출물은 Git에 포함되지 않는다. Drive 공유 후 파일명과 SHA-256을 함께 관리하고, 전달 과정에서 해시가 달라지면 사용을 중단해야 한다.
6. 실제 팀 공유 Drive 경로가 제공되면 공개 링크 대신 접근 제한된 내부 경로를 작업 인수인계 문서에 추가해야 한다.
