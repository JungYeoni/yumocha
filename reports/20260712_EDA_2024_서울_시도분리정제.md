# [EDA] 2024년 서울특별시 세부사업 예산현황 분리·정제 — 2026-07-12

이슈 #3 1차 샘플 작업 결과. 원본: `notebooks/20260712_EDA_2024_서울_시도분리정제.ipynb`
전체 작업계획: `reports/20260712_시도별_예산현황_분리정제_작업계획.md`

## 분석 목적

`data/raw/`의 2024년도 시행계획 예산현황 원본(17개 시도가 한 시트에 이어붙은 구조)에서 **서울특별시** 구간만 분리해, 시도별 정제 파이프라인(경계탐지 → 헤더 평탄화 → 계층 태깅 → leaf/long 변환 → QA 대조)이 실제로 동작하는지 1차 검증한다.

## 처리 파이프라인

1. `find_sido_boundaries`로 `붙임 (OO시 OO교육청)` 타이틀 텍스트를 정규식으로 스캔해 17개 시도 경계(시작/끝 행) 확보
2. `slice_sido`로 서울 구간(원본 0~285행, 285행)만 슬라이싱
3. `flatten_header`로 2단 병합헤더를 단일 컬럼명으로 평탄화, 전부 NaN인 병합 잔여 컬럼 제거
4. 컬럼명·텍스트 값(`세부사업명`, `공통/자체`, `주요내용`)의 개행·중복공백 정제
5. `classify_row`로 각 행을 `대분류_소계` / `중분류_소계` / `헤더반복` / `세부사업(leaf)`로 분류 후, `대분류`·`중분류`를 ffill로 하위 행에 전파
6. 소계·헤더반복 행 제거 → leaf 테이블(`df_leaf`) 확보
7. `find_budget_columns`로 연도 컬럼을 자동판별해 `예산_당해년도`/`예산_전년도_최종`으로 표준화
8. `df_leaf.melt()`로 long 포맷(`df_long`) 생성
9. leaf를 `(대분류, 중분류)` 기준 합산해 원본 중분류 소계 행 값과 QA 대조
10. `data/interim/`에 저장

## 처리 중 발견한 이슈 3건

| 이슈 | 원인 | 해결 |
|------|------|------|
| 반복 헤더 행이 leaf로 잘못 분류됨 | 원본 PDF가 페이지마다 표 헤더를 다시 찍는 방식이었는데, 엑셀 변환 시 그 반복 헤더 줄이 본문 중간에도 그대로 남아있음 (`세부사업명`, `총예산(a)` 같은 헤더 텍스트가 데이터 행에 등장) | `classify_row`에 "이름이 헤더 텍스트와 같거나 비어있으면 헤더반복" 조건 추가 |
| 중분류 소계 행 1건이 leaf로 잘못 분류됨 | 해당 행의 `주요내용` 셀에 보이지 않는 유니코드 특수문자(``)가 남아있어 "주요내용이 비어있어야 중분류_소계"라는 판별 조건을 통과하지 못함. 그 여파로 뒤따르는 leaf 행들까지 잘못된 중분류로 ffill됨 | 판별 조건에서 `주요내용 비어있음` 요건 제거, `숫자.`로 시작하는지만으로 판별 |
| 조건문 논리 오류(`!=` / `==` 반전) | `classify_row` 수정 중 조건이 반대로 들어가 사실상 모든 행이 "헤더반복"으로 분류됨 | 조건 반전 수정 후 커널 재시작·전체 재실행으로 상태 정합성 확보 |

## 분석 결과

**핵심 발견:**
- 서울 구간(285행)에서 실제 세부사업(leaf) **237건** 추출, 대분류는 `Ⅰ. 공통사업`(47건) / `Ⅱ. 지자체 자체사업`(190건) 2개
- leaf 예산 합계가 대분류 소계 행 값과 정확히 일치 (`3,906,937 + 965,746 = 4,872,683`)
- QA 대조(중분류별 leaf 합계 vs 원본 소계 행) **불일치 0건** — 전부 일치

**수치 요약:**

| 지표 | 값 |
|------|-----|
| 서울 구간 원본 행 수 | 285 |
| leaf(세부사업) 건수 | 237 |
| long 포맷 행 수 | 474 (237 × 2개년) |
| 대분류 개수 | 2 (공통사업, 지자체 자체사업) |
| 서울 전체 예산(2024년) | 4,872,683 |
| QA 불일치 건수 | 0 |

**해석:** 텍스트 기반 시도 경계 탐지, 2단 헤더 평탄화, 소계/leaf 계층 분리 로직이 2024년 서울 데이터에 대해 정확하게 동작함을 확인했다. 다만 원본이 PDF→엑셀 변환 산출물이라 반복 헤더·비가시 특수문자 같은 노이즈가 산발적으로 섞여 있어, 규칙 기반 분류 로직은 예외 케이스를 하나씩 발견하며 보강하는 과정이 필요했다.
