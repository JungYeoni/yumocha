## 1. 배경

- `data/raw/`에는 연도별 파일 9개(2016~2024, 제3차/제4차 기본계획)가 있으며, 시도별 파일 분리는 없다.
- 파일마다 시트는 `Table 1` 하나뿐이며, 그 안에 17개 시도가 정해진 순서로 이어 붙어 있는 단일 롱 테이블 형태다 (총 7,844~13,014행).
- 시도 순서는 고정되어 있다: 서울 → 부산 → 대구 → 인천 → 광주 → 대전 → 울산 → 세종 → 경기 → 강원 → 충북 → 충남 → 전북 → 전남 → 경북 → 경남 → 제주.
- 소관기관(교육청) 정보는 시도 타이틀 행에 `붙임 (OO시 OO시교육청)` 형태로 함께 기재되어 있으며, 별도 컬럼으로는 존재하지 않는다.

## 2. 핵심 결론 (요약)

- 데이터수집팀이 `data/raw/칼럼정렬/`에 연도별로 `정리본_자동`이라는 사전 가공 시트를 마련했다. 시도 구분(`지역` 컬럼)과 컬럼 구조 통일이 이미 되어 있어, 이를 1차 입력(권장 경로)으로 사용하면 시도 경계 탐지·OCR·헤더 평탄화 단계를 생략할 수 있다.
- 대분류/중분류 계층 판별(`classify_row`)과 QA 대조(leaf 합계 vs 소계값)는 `정리본_자동`을 사용하더라도 동일하게 필요하다.
- `정리본_자동`의 `행유형="연속행 후보"` 태그는 실제 페이지 잘림과 VBA 오탐이 혼재되어 있어 그대로 신뢰할 수 없으며, 표본 검증이 필요하다.
- 원본(`Table 1`)을 직접 파싱하는 기존 경로(시도 경계 탐지 + OCR)는 폐기하지 않고 대안·교차검증용으로 유지한다. 2024년 서울 표본은 이 경로로 검증을 완료했다.
- 2016~2019년의 `계/국비/지방비` 구조는 `계` 값만 leaf 예산으로 사용하고 국비/지방비 행은 제외하는 것으로 결정했다.

## 3. 데이터 구조 조사 결과

### 3-1. `정리본_자동` 시트 — 시도 분리·컬럼 통일이 완료된 데이터

`data/raw/칼럼정렬/` 폴더에 연도별 파일 9개(`{연도}..._칼럼정렬.xlsx`)가 있다. 데이터수집팀이 GPT로 VBA를 작성해 병합 셀 위치를 연도별로 통일한 결과물이다. `Table 1` 시트만 놓고 보면 원본과 값이 완전히 동일하지만(값 diff 0건), 이는 `Table 1`이 원본을 그대로 보존해둔 시트이기 때문이며, 실제 가공 결과물은 별도 시트에 있다.

| 시트명 | 내용 |

시트 구성, 컬럼명, `지역`(17개 시도 전체 포함)은 9개 연도 전부 동일하다. 원본 대비 행 수는 다소 줄어 있으며(예: 2017년 원본 13,014행 → 정리본 11,808행), 이는 노이즈가 일부 제거된 결과로 판단된다.

**2022년 최상위 폴더 특이사항 (주의)**: `data/raw/`(`칼럼정렬/` 폴더 밖) 최상위에도 연도별 원본 파일이 있는데, 2022년 파일만 유일하게 `원본`/`정리본_전체`/`정리본_사업만`/`검증` 4개 시트가 이미 붙어 있다. 다른 연도의 최상위 파일은 `Table 1` 하나뿐이다(원본 그대로). 이 2022년 전용 가공 시트는 컬럼명이 `사업분류`/`원본페이지`/`행구분`/`병합원본행` 등으로 표준 `정리본_자동`과 스키마가 다르며, 표준 VBA 파이프라인(위 §3-1)을 만들기 전에 2022년만 먼저 별도 수작업/스크립트로 시험 처리해본 산출물로 추정된다(확인된 사실은 아니고 정황상 추정). **표준 입력은 어디까지나 `data/raw/칼럼정렬/2022..._칼럼정렬.xlsx`의 `정리본_자동` 시트이며(다른 연도와 동일하게 정상 존재함을 확인함), 최상위 파일의 `정리본_전체`/`정리본_사업만` 시트는 사용하지 않는다.** 2022년 담당자에게는 이 점을 명시적으로 안내해 혼동을 방지한다.

**`행유형` 값의 의미**: 본 계획에서 사용하는 `구분`(대분류_소계/중분류_소계/세부사업/헤더반복) 컬럼과 역할은 유사하나 판별 기준이 다르다. `구분`은 해당 행의 계층적 위치를 나누고, `행유형`은 해당 행 내용의 완결성을 나눈다.

| 값 | 의미 |
|---|---|
| `데이터행` | 정상 행 — 세부사업명·예산·주요내용 등이 제대로 채워짐. 단, 대분류_소계·중분류_소계·leaf가 전부 이 값으로 묶여 있어 계층 구분은 제공하지 않음 |
| `주요내용 연속행 후보` | 세부사업명 등은 비어있고 `주요내용`만 남은 행 — 앞 행 설명이 페이지 경계에서 잘려 이어진 조각일 가능성을 VBA가 표시한 것 |
| `구분행사업명 연속 후보` | `세부사업명` 자체가 짧게 끊긴 것으로 보이는 행 — 이름이 앞/뒤 행과 이어져야 할 가능성을 표시한 것 |
| `NaN` | 판별 대상에서 제외된 행. 확인 결과 2016~2020년에서 각 연도당 정확히 1건씩이며, 항상 `(단위 : 백만원)` 문서 상단 단위 표기 행이었다. 페이지 잘림이나 소계와 무관한 문서 제목 잔재다 |

`행유형` 태그는 그대로 신뢰할 수 없다. 2021년 데이터를 확인한 결과, `주요내용 연속행 후보`로 태깅된 행 중 다수가 실제 페이지 잘림이 아니라 `세부사업명=NaN`, `주요내용="1  서울특별시"`처럼 VBA가 페이지 타이틀 등 애매한 행을 넓게 잡아낸 사례였다. "후보" 태그는 추가 검증이 필요하다는 신호로만 사용하고, 최종 판단은 QA(leaf 합계 vs 소계값 대조)로 내린다.

`정리본_자동`이 대체하지 못하는 부분은 계층 정보다. 대분류/중분류 정보는 여전히 `세부사업명` 텍스트 안에 `"Ⅰ. 공통사업"`, `"1. 함께 일하고..."` 형태로만 남아 있고, `행유형`도 이를 구분해주지 않는다. 따라서 `classify_row`와 QA 대조 로직은 `정리본_자동` 사용 여부와 무관하게 그대로 필요하다.

### 3-2. 원본(`Table 1`) 구조 — 대안 경로 참고자료

`정리본_자동`이 없거나 신뢰할 수 없는 경우 사용하는 대안 경로(§4-3)를 위해, 원본을 직접 파싱하며 확인한 내용을 정리한다. 2024년 서울 표본은 이 경로로 검증을 완료했다.

#### 3-2-1. 시도 경계 타이틀: 텍스트 vs 이미지 (+ OCR)

같은 형식의 파일이라도 연도에 따라 시도 구분 타이틀이 텍스트로 존재하는 경우와 이미지로만 존재하는 경우가 혼재한다.

| 연도 | 총 행수 | 임베디드 이미지 수 | 시도 타이틀 텍스트 검출 | 비고 |
|---|---|---|---|---|
| 2016 | 7,844 | 314 | 11/17 (서울·대구·광주·울산·세종·충남 누락) | 이미지가 본문 셀에도 다수 섞여 있어 타이틀 이미지만 특정하기 어려움 |
| 2017 | 13,014 | 333 | 13/17 (서울·대구·인천·대전·세종·충남 누락) | 상동 |
| 2018 | 9,301 | 829 | 17/17 | 이미지 개수는 많지만 타이틀은 텍스트로 존재 |
| 2019 | 9,804 | 17 | 12/17 (서울·대구·인천·세종·충남 누락) | 이미지 17개 = 시도 수 17개, 위치가 섹션 시작과 정확히 일치 → 타이틀이 전부 이미지 |
| 2020 | 7,904 | 17 | 13/17 | 2019와 동일 패턴 |
| 2021 | 10,672 | 468 | 17/17 | |
| 2022 | 9,751 | 181 | 17/17 | |
| 2023 | 8,723 | 17 | 12/17 (서울·대구·광주·울산·세종 누락) | 2019/2020과 동일 패턴 |
| 2024 | 8,234 | 87 | 17/17 | `붙임 (OO시 OO시교육청)` 정규식으로 전부 매칭, 구조가 가장 단순 |

이미지 내용은 openpyxl의 셀 텍스트 추출로는 읽을 수 없으나, OCR을 적용하면 코드로 자동화할 수 있다. 2016년 파일에서 텍스트로 누락됐던 6건을 `tesseract` + `kor` 언어팩으로 검증한 결과는 다음과 같다.

| 시도 | 확인된 행 | OCR 원본 결과 | 17개 후보 매칭 |
|---|---|---|---|
| 서울특별시 | ~61 | `서울특별시` | 일치 |
| 대구광역시 | 1,115 | (육안 확인, OCR 재검증 생략) | 일치 |
| 광주광역시 | 1,660 | `광주광역시` | 일치 |
| 울산광역시 | 2,686 | `올산광역시` (한 글자 오독) | 편집거리 최소값으로 자동 보정 → 울산광역시 |
| 세종특별자치시 | 2,994 | `세종특별자치시` | 일치 |
| 충청남도 | 4,594 | `충청남도` | 일치 |

6건 중 5건은 OCR이 정확히 일치했고, 1건(울산)은 한 글자를 오독했다. 정답이 17개 시도명으로 정해진 폐쇄형(closed-set) 문제이므로, OCR 결과와 17개 후보 사이 편집거리(fuzzy match)를 계산해 가장 가까운 후보로 자동 보정하면 실질적으로 100%에 가까운 정확도를 확보할 수 있다. 전 건이 표준 행정구역 순서와도 일치했다.

참고로 이미지 안의 반복 이미지 대부분(2016년 314건 중 다수)은 "Bridge Plan 2020" 워터마크와 "2016년 지방자치단체 시행계획" 페이지 헤더로 확인됐다. PDF→엑셀 변환 과정에서 매 페이지 삽입된 장식 이미지이며, 항상 1번째 열(빈 칸)에만 위치해 데이터 텍스트와 겹치지 않으므로 텍스트 읽기에는 영향이 없다. 시도명 타이틀 배너는 이 워터마크와 크기가 달라 구분이 가능했다.

**OCR 처리 절차:**

1. 파일별 임베디드 이미지의 바이트 크기를 집계해 최빈 크기(워터마크)를 제외
2. 남은 후보 중 각 시도 섹션 시작 지점 근처 이미지를 추출
3. `pytesseract`로 OCR 실행
4. OCR 결과를 17개 시도명 후보와 편집거리로 비교해 가장 가까운 후보로 확정 (`rapidfuzz` 사용)
5. 편집거리가 임계값 이상인 경우에만 검수 대상으로 표시

**필요 환경**: `tesseract`(Homebrew: `brew install tesseract tesseract-lang`, Linux: `apt install tesseract-ocr tesseract-ocr-kor`), Python `pytesseract`, `rapidfuzz` (프로젝트 의존성에 추가 완료).

#### 3-2-2. 헤더/컬럼 구조의 연도별 차이

`skiprows`는 `pandas.read_excel()`이 표를 읽을 때 상단에서 건너뛸 행 수를 지정하는 인자다. 원본 파일은 실제 헤더 위에 "붙임 (OO시 OO교육청)", "(단위: 백만원)" 같은 타이틀/부제목 행이 있으며, 이 행 수가 연도마다 달라 `skiprows` 값도 파일마다 다르다.

| 연도 | skiprows | 헤더 행수 | 특이사항 |
|---|---|---|---|
| 2016~2019 | 1 | 2행 | `재정 구분` 컬럼에 `계/국비/지방비`가 있어 세부사업 1개당 3행으로 분리됨. `공통/자체` 컬럼 없음 |
| 2020 | 2 | 2행 | 참조 함수와 거의 일치 |
| 2021 | 3 | 2행 | 타이틀 위에 부제목 행이 하나 더 있어 skiprows가 밀림 |
| 2022 | 1 | 2행 | |
| 2023 | 3 | 2행 | 컬럼명 자체에 시도명이 붙는 경우 있음(예: `세부사업명 (서울)`) → 컬럼명 정규화 필요 |
| 2024 | 2 | 2행 | 참조 함수 가정과 정확히 일치 |

`skiprows`는 파일마다 다르므로 고정값을 사용할 수 없으며, 파일별 자동 탐지 로직이 필요하다.

#### 3-2-3. (a)/(b) 연도 라벨 자동판별

`find_budget_columns` 로직(컬럼명에서 4자리 연도를 추출해 큰 값을 당해년도, 작은 값을 전년도로 판별)은 유효함을 확인했다. 2016~2019년 파일은 `(A)/(B)` 라벨, 2022~2024년은 `(a)/(b)` 라벨을 사용하나 동일 로직으로 처리 가능하다.

#### 3-2-4. 중분류 번호 체계 및 연도별 표 형식 차이 (데이터수집팀 교차검증)

데이터수집팀(최연호)이 확인한 내용에 따르면, 같은 중분류 번호(예: `1. 함께 일하고 돌보는 사회`)가 대분류가 다르면 각각 별도로 존재한다 — 공통사업 쪽에도 `1. 함께 일하고 돌보는 사회`가 있고, 자체사업 쪽에도 같은 번호가 별도로 있다.

이는 본 계획의 설계 방침과 일치한다. `중분류` 값은 번호를 파싱해 비교하지 않고 문자열 전체를 라벨로 취급해 행 순서대로 ffill로 채우며, QA 대조도 `groupby(["대분류", "중분류"])`로 대분류까지 함께 묶어 집계한다. 따라서 대분류가 다른 두 "1. 함께 일하고 돌보는 사회"가 서로 섞여 합산되지 않는다.

같은 확인 과정에서 제3차(2016~2019)와 제4차(2020~2024) 기본계획 시기별로 표 형식이 다르다는 점도 교차검증됐다. 이는 §3-2-2에서 파악한 `계/국비/지방비` 구조 차이와 일치한다.

## 4. 처리 파이프라인

### 4-1. 범위 (Scope)

"1개 연도, 그 연도의 여러 시도" 시나리오는 어느 연도를 선택해도 가능함을 확인했다(파일 하나에 이미 17개 시도가 포함되어 있음). 검증용 표본은 다음과 같이 진행한다.

1. 2024년 · 서울/부산 — `정리본_자동` 경로로 우선 검증 (서울은 원본 파싱 경로로 검증 완료, 부산으로 `정리본_자동` 경로 자체를 검증)
2. 2019년 · 서울 — `정리본_자동`이 있어도 신뢰도가 낮을 수 있는 이미지 전용 시도명 케이스와 비교 대조
3. (2차 범위) 2016~2019년의 `계/국비/지방비` 3행 구조 — `정리본_자동`이 이 구조를 어떻게 처리했는지 확인 필요

### 4-2. 실행 순서 — 권장 경로 (`정리본_자동` 활용)

1. `정리본_자동` 시트 로드 — `pd.read_excel(path, sheet_name="정리본_자동", header=0)`. skiprows 자동탐지·헤더 평탄화 불필요
2. 시도별 필터링 — `지역` 컬럼으로 분리 (경계 탐지·OCR 불필요)
3. 계층 태깅 — `classify_row`로 대분류_소계/중분류_소계/헤더반복/세부사업 판별 후 대분류/중분류 ffill (§4-4 기존 함수 재사용)
4. `행유형` 교차검증 — "연속행 후보" 태깅된 행을 확인해 실제 페이지 잘림인지 VBA 오탐인지 판별 (§3-1 참고, 태그를 그대로 신뢰하지 않음)
5. leaf/long 변환
6. QA 대조 — leaf 합계 vs 소계 행 값 검증
7. 원본(`Table 1`) 대조 — 동일 시도·연도를 §4-3 경로로도 처리해 결과 일치 여부를 표본 검증 (최소 1개 시도는 두 경로 모두 확인)
8. 저장 — `data/interim/{시도명}/`에 wide·long·필터링전 전체원본, `reports/`에 QA 검증결과

### 4-3. 실행 순서 — 대안 경로 (원본 `Table 1` 직접 파싱)

`정리본_자동`이 없는 연도가 확인되거나, 4-2의 7번 교차검증에서 불일치가 발견될 경우 사용한다. 2024년 서울 표본에서 검증을 완료한 경로다.

1. skiprows 자동탐지 — 파일별 헤더 시작 행 확인
2. 시도 경계 찾기 — 텍스트 정규식 우선, 텍스트가 없는 구간은 OCR + 17개 후보 편집거리 매칭으로 보완 (§3-2-1)
3. 시도별 슬라이싱 — 확정된 경계 기준으로 시트를 시도별 원본 조각(raw slice)으로 분리
4. 조각별 정제 — 헤더 평탄화 → 계층 태깅 → 대분류/중분류 ffill → leaf/long 변환 (§4-4 참조 함수 재사용)
5. QA 대조
6. 저장

### 4-5. 연도 확장 체크리스트

2024년(17개 시도, #5)에서 검증된 파이프라인을 다른 연도에 적용하기 전에, 아래 항목을 먼저 확인한다 — 연도마다 원본 문서 작성 방식이 달라 그대로 안 통할 수 있다.

| 확인 항목 | 확인 방법 | 다르면 해야 할 일 |
|---|---|---|
| `정리본_자동` 컬럼명이 동일한가 | `df.columns.tolist()`로 확인 | 다르면 `find_budget_columns`로 당해/전년 컬럼을 패턴 탐지 (하드코딩 금지) |
| 대분류/중분류 헤더 패턴이 동일한가 | `classify_row` 적용 후 시도별 `대분류_소계`/`중분류_소계` 개수가 일관적인지(`groupby(["지역","사업행구분"]).size()`) 확인 | 개수가 어긋나는 시도가 있으면 오분류 사례(#5의 경기·경남처럼 숫자로 시작하는 세부사업명)를 직접 찾아 조건 보강 |
| "붙임 (OO시 OO교육청)" 같은 시도경계 노이즈 잔존 여부 | `헤더반복` 분류 개수가 시도당 0~1건인지 확인 | 다른 패턴의 노이즈면 `sido_title_pattern` 정규식 보강 |
| 증감액/비율 부호 소실 여부 | 당해-전년 재계산 값과 원본 부호 대조 (#5 방법 재사용) | 있으면 무조건 재계산 값 사용, 원본 값은 버림 |
| 2016~2019년 `계/국비/지방비` 3행 구조 해당 여부 | 연도 확인만으로 판별 가능(§3-2-2) | 해당 연도는 재정구분="계" 행만 남기는 별도 필터 적용 (§2 결정사항 참조), 위 체크리스트는 그 이후에 적용 |

확인이 끝난 함수만 연도 파라미터를 받는 형태로 정리해 재사용한다. 시도명 매핑(단축명/전체명/영문명)은 `data/lookup/시도_지역코드_매핑.csv`를 참조한다.

### 4-6. 연도 간 예산 비교 전 다중 행 처리

연도 간 본예산·최종예산을 비교할 때 `지역+세부사업명`만으로 병합하면 공통/자체가 다른 사업이 섞일 수 있다. 공통/자체를 추가해도 같은 키가 한 연도에 여러 행이면 pandas `merge`가 카테시안 곱을 만들어 서로 다른 구성 항목의 예산을 잘못 비교한다.

2023/2024 부산 `노인복지관 운영 지원(자체)`에서 각 연도의 일반 운영지원과 셔틀버스 지원 행이 2×2로 연결되어, 23과 23,655의 비교에서 102,747.8%라는 가짜 극단값이 생성된 사례가 확인됐다. 두 행을 사업명 단위로 합산하면 17,276 대 23,678로, 변동률은 약 37.1%이다.

후속 QA에서 경기·강원 등은 서로 다른 시군구가 같은 세부사업명을 사용하는 사례가 다수 확인되었다. 본 프로젝트의 분석 단위는 시도가므로, 이를 시군구별로 분리하지 않고 같은 시도의 동명 사업을 시도 단위 사업군으로 합산한다. 다만 구성 차이를 잃지 않도록 주요내용 목록과 구성항목 수를 함께 보존한다.

**표준 처리 순서:**

1. 표시용 사업명은 개행·탭·연속 공백을 한 칸으로 정리하고, 별도로 내부 공백을 제거한 사업명 매칭 키를 만든다.
2. 시도 단위 매칭 키를 `지역+사업분류재정구분+세부사업명_공백제거`로 구성한다.
3. 각 연도 안에서 키별 행 수를 집계해 2건 이상인 키를 구성항목이 여러 개인 사업군으로 기록한다.
4. 병합 전에 키별 예산액을 합산한다. 대표 사업명, 원본 사업명 목록, 주요내용 목록과 `구성항목수`를 보존한다.
5. 집계된 두 연도 테이블을 같은 키로 `merge(..., validate="one_to_one")`한다. 일대일 조건을 만족하지 않으면 즉시 오류를 발생시켜 잘못된 변동률 산출을 차단한다.
6. 합산 단위 변동률을 기본 예산 지표로 사용하고, 문서 간 구성항목 수 또는 주요내용 목록이 달라진 경우 별도 검토 플래그를 남긴다.

**필수 QA:** 병합 전 연도별 키 수·중복 키 수·최대 중복 행 수, 집계 전후 예산 총액 보존, 병합 후 행 수, 미매칭 키 수를 기록한다. 이 절차를 통과하지 않은 변동률·그래프·통계검정 결과는 최종 산출물로 사용하지 않는다.

**공백 표기 QA:** 2023년 자료에서 `지역+사업분류재정구분+공백제거 사업명`이 같지만 원래 표기가 두 종류 이상인 그룹 79개가 확인되었고, 모두 당해·전년도 예산이 달랐다. 시도 단위 분석에서는 이를 오류 중복이 아니라 여러 구성행을 가진 동명 사업군으로 해석해 예산을 합산한다. 단, 원본 사업명·주요내용·구성항목 수는 QA와 연도 간 구성 변화 확인을 위해 보존한다.

## 5. 산출물 (Deliverables)

본 저장소는 `README.md`에 정의된 `data/raw/` → `data/interim/` → `data/processed/` 3단 폴더 구조를 따른다. 임의로 새 폴더를 만들거나 `data/raw/`의 원본을 직접 수정·덮어쓰기 하지 않는다.

- **`data/raw/`**: 원본 엑셀 9개, 읽기 전용으로만 사용 (git 추적 제외 대상이므로 이 안에서 새로 만들지 않음)
- **`data/interim/`**: 시도별 결과는 `data/interim/{시도명}/` 하위 폴더에 모은다. 폴더로 시도가 구분되어도 파일명에 시도명을 함께 표기해, 파일이 폴더 밖으로 이동되거나 검색될 때도 식별 가능하도록 한다.
  - 시도 경계 탐지 결과 요약표(대안 경로 전용, 연도 전체 17개 시도를 함께 다룸): `data/interim/{연도}_시도경계_탐지로그.csv`
  - 시도별 wide(leaf) 정제 테이블: `data/interim/{시도명}/{연도}_{시도명}_세부사업_정제.csv` — 최종 컬럼 스키마는 아래 표 참조 (2024년 17개 시도분 생성 완료, #5)
  - 시도별 long(tidy) 테이블: `data/interim/{시도명}/{연도}_{시도명}_세부사업_정제_long.csv` — **미착수, 후속 작업**
  - 필터링 전 시도별 전체 원본(소계·헤더반복 노이즈까지 포함, QA용 보관): `data/interim/{시도명}/{연도}_{시도명}_필터링전_전체원본.csv` (2024년 17개 시도분 생성 완료, #5)
- **`data/lookup/`**: 시도명 매핑 등 연도·작업에 무관하게 재사용하는 고정 참조 테이블
  - `시도_지역코드_매핑.csv`: 지역(단축명)/지역명_전체/지역명_영문/지역코드 (지역코드는 행정안전부 법정동코드 앞 2자리, 시도 단위 기준으로 확정 완료 — 서울 11 ~ 제주 50. 출처: `code.go.kr`)
- **`data/processed/`**: 현재 단계에서는 사용하지 않는다. 17개 시도 × 여러 연도 처리가 끝난 뒤, 전부 이어붙인 최종 통합 테이블을 생성할 때 이 폴더에 저장한다. 시도별 개별 결과물은 그 통합 테이블을 만들기 위한 중간 산출물이므로 `data/interim/`에 둔다.
- **`reports/`**: 팀 공유용 QA 결과와 본 계획 문서 등 리포트류
  - QA 대조 결과: `reports/{연도}_전국_QA_검증결과.csv` — **연도 단위로 17개 시도를 한 파일에 합쳐서 저장한다** (시도별로 나누면 시도 간 비교를 통한 이상치 발견이 어려워지므로, #5에서 시도별 대신 전국 통합 파일로 확정). 검증 대상 전체와 일치/불일치 결과 컬럼을 항상 저장 — 불일치가 0건이어도 검증 건수가 파일 자체에 남도록, 불일치 행만 걸러서 저장하지 않음
