# 시행계획 세부사업 예산현황 — 시도별 분리·정제 작업계획 — 2026-07-12

작성: Claude Code / 대상: 유모차팀
원본 파일을 직접 열어 확인한 실제 구조를 반영해 작성함.

---

## 1. 배경

- `data/raw/`에는 **연도별 파일 9개**(2016~2024, 제3차/제4차 기본계획)가 있고, 시도별 파일 분리는 없음.
- 파일마다 시트는 `Table 1` 하나뿐이며, 그 안에 **17개 시도가 정해진 순서로 이어 붙어 있는 단일 롱 테이블**이다 (총 7,844~13,014행).
- 시도 순서는 고정: 서울 → 부산 → 대구 → 인천 → 광주 → 대전 → 울산 → 세종 → 경기 → 강원 → 충북 → 충남 → 전북 → 전남 → 경북 → 경남 → 제주.
- 소관기관(교육청) 정보는 시도 타이틀 행에 `붙임 (OO시 OO시교육청)` 형태로 같이 붙어 있고, 별도 컬럼으로 존재하지 않는다.

---

## 2. 핵심 결론 (요약)

- 데이터수집팀이 `data/raw/칼럼정렬/`에 연도별로 **`정리본_자동`이라는 사전 가공 시트**를 만들어뒀다. 시도 구분(`지역` 컬럼)과 컬럼 구조 통일이 이미 되어 있어, 이걸 **1차 입력(권장 경로)**으로 쓰면 시도 경계 탐지·OCR·헤더 평탄화 단계를 전부 생략할 수 있다.
- 다만 대분류/중분류 계층 판별(`classify_row`)과 QA 대조(leaf 합계 vs 소계값)는 **정리본_자동을 쓰더라도 그대로 필요**하다.
- `정리본_자동`의 `행유형="연속행 후보"` 태그는 실제 페이지 잘림과 VBA 오탐이 섞여 있어 **그대로 신뢰하면 안 되고** 표본 검증이 필요하다.
- 원본(`Table 1`)을 직접 파싱하는 기존 경로(시도 경계 탐지 + OCR)는 **폐기하지 않고 대안/교차검증용으로 유지**한다 — 2024년 서울 샘플은 이미 이 경로로 검증 완료됨.

## 3. 데이터 구조 조사 결과

`data/raw/칼럼정렬/` 폴더에 연도별 파일 9개(`{연도}..._칼럼정렬.xlsx`)가 있다. 데이터수집팀이 GPT로 VBA를 작성해 병합 셀 위치를 연도별로 통일한 결과물이다. 파일을 열어보면 원본과 완전히 동일해 보이지만(값 diff 0건), 이는 시트 `Table 1`(원본 그대로 보존)만 비교했기 때문이고, **실제 가공 결과물은 별도 시트에 있다.**

| 시트명 | 내용 |
|---|---|
| `Table 1` | 원본 그대로 (기존 `data/raw/` 파일과 100% 동일, 대조용으로 보존) |
| `정리본_자동` | 실제 가공 결과물. 아래 상세 |
| `검증_자동` | 원본표구간(페이지/블록 단위)별로 셀 범위를 어떤 컬럼에 매핑했는지 남긴 로그. 2021년 기준 306개 블록, 전부 `점검내용=정상` |

**컬럼 구조** (12개, 9개 연도 전부 동일): `지역 | 세부사업명 | 사업분류재정구분 | {당해}년 예산 | {전년}년 예산 | 증감액 | 비율 | 주요내용 | 원본표구간 | 머리글행 | 원본행 | 행유형`

시트 구성·컬럼명·`지역`(17개 시도 전부 포함)은 9개 연도 전부 동일하다. 원본 대비 행 수가 꽤 줄어있는데(예: 2017년 원본 13,014행 → 정리본 11,808행), 노이즈가 이미 일부 제거된 것으로 보인다.

**`행유형` 값의 뜻** — 저희가 만든 `구분`(대분류_소계/중분류_소계/세부사업/헤더반복) 컬럼과 역할은 비슷하지만 기준이 다르다. `구분`은 "이 행이 계층상 뭐냐"를 나누고, `행유형`은 "이 행 내용이 온전하냐"를 나눈다.

| 값 | 뜻 |
|---|---|
| `데이터행` | 정상 행 — 세부사업명·예산·주요내용 등이 제대로 채워짐. **단, 대분류_소계·중분류_소계·leaf가 전부 이 값으로 뭉쳐 있어 계층 구분은 안 해줌** |
| `주요내용 연속행 후보` | 세부사업명 등은 비어있고 `주요내용`만 남은 행 — 앞 행 설명이 페이지 경계에서 잘려 이어진 조각일 수 있다는 VBA의 추정 |
| `구분행사업명 연속 후보` | `세부사업명` 자체가 짧게 끊긴 것처럼 보이는 행 — 이름이 앞/뒤 행과 이어져야 할 수 있다는 추정 |
| `NaN` | 판단 자체를 안 한 행. **직접 확인한 결과, 2016~2020년 전부 정확히 1건씩이고 항상 `(단위 : 백만원)`이라는 문서 맨 위 단위 표기 줄이었다** — 페이지 잘림도 소계도 아닌 무해한 문서 제목 잔재 |

**주의: `행유형` 태그를 그대로 신뢰하면 안 된다.** 2021년 데이터를 직접 까본 결과, `주요내용 연속행 후보`로 태깅된 행 중 다수가 실제 페이지 잘림이 아니라 `세부사업명=NaN`, `주요내용="1  서울특별시"`처럼 **VBA가 애매한 행(페이지 타이틀 근처 등)을 넓게 잡아낸 것**이었다. "후보"는 사람/추가 검증이 필요하다는 신호로만 쓰고, 최종 판단은 저희 쪽 QA(leaf 합계 vs 소계값 대조)로 한다.

**`정리본_자동`이 대체하지 못하는 것**: 대분류/중분류 계층 정보는 여전히 `세부사업명` 텍스트 안에 `"Ⅰ. 공통사업"`, `"1. 함께 일하고..."` 형태로만 남아있고, `행유형`도 이를 구분해주지 않는다. 즉 **`classify_row`와 QA 대조 로직은 그대로 필요**하다.

#### VBA(.bas) 코드 검토 결과 — `data/raw/칼럼정렬/2016-2025_표칼럼_범용정렬_폴더오류수정.bas`

데이터수집팀이 GPT로 작성한 VBA 원본을 직접 읽어봤다. 한글 문자열 리터럴은 인코딩이 깨져서(`�`로 표시, EUC-KR로 저장된 걸 다른 인코딩으로 열어본 탓으로 추정) 안 보이지만, **변수명·함수 구조·제어 흐름은 전부 ASCII라 로직 자체는 완전히 읽고 이해할 수 있다.** 블랙박스 로직은 없고, 표준적인 패턴 매칭 + 엑셀 Range 조작뿐이다.

**핵심 로직 (함수별):**

| 함수 | 역할 |
|---|---|
| `NormalizeWorkbookCore` | 메인 로직. 아래 단계를 순서대로 수행 |

**예상 못한 발견 — `RemoveNationalAndLocalRows`**: 자동 파이프라인에는 안 물려있는 **별도 수동 유틸리티 함수**로, 2016~2019년 `계/국비/지방비` 구조 전용이다. 재정구분이 "계" 또는 "국비"인 행을 삭제해서 지방비 행만 남기는 로직 — §7 결정사항의 "계/국비/지방비를 어떻게 다룰지" 질문에 대해 데이터수집팀이 이미 만들어둔 답이다. 다만 자동 실행되지 않고 사람이 직접 돌려야 하는 함수라는 점은 유의해야 한다.

### 3-2. 원본(`Table 1`) 구조 — 대안 경로 참고용

`정리본_자동`이 없거나 신뢰할 수 없을 때 쓰는 대안 경로(§4-3B)를 위해, 원본을 직접 파싱하며 확인한 내용을 정리해둔다. 2024년 서울 샘플은 이 경로로 이미 전부 검증 완료했다.

#### 3-2-1. 시도 경계 타이틀: 텍스트 vs 이미지 (+ OCR)

같은 형식의 파일이라도 연도에 따라 시도 구분 타이틀이 **텍스트로 존재하는 경우**와 **이미지로만 존재하는 경우**가 섞여 있다.

| 연도 | 총 행수 | 임베디드 이미지 수 | 시도 타이틀 텍스트 검출 | 비고 |
|---|---|---|---|---|
| 2016 | 7,844 | 314 | 11/17 (서울·대구·광주·울산·세종·충남 누락) | 이미지가 본문 셀에도 다수 섞여 있어 타이틀 이미지만 특정하기 어려움 |
| 2017 | 13,014 | 333 | 13/17 (서울·대구·인천·대전·세종·충남 누락) | 상동 |
| 2018 | 9,301 | 829 | **17/17** | 이미지 개수는 많지만 타이틀은 텍스트로 존재 |
| 2019 | 9,804 | **17** | 12/17 (서울·대구·인천·세종·충남 누락) | 이미지 17개 = 시도 수 17개, 위치가 섹션 시작과 정확히 일치 → **타이틀이 전부 이미지** |
| 2020 | 7,904 | **17** | 13/17 | 2019와 동일 패턴 |
| 2021 | 10,672 | 468 | **17/17** | |
| 2022 | 9,751 | 181 | **17/17** | |
| 2023 | 8,723 | **17** | 12/17 (서울·대구·광주·울산·세종 누락) | 2019/2020과 동일 패턴 |
| 2024 | 8,234 | 87 | **17/17** | `붙임 (OO시 OO시교육청)` 정규식으로 전부 매칭, 구조 가장 깔끔 |

이미지 내용은 openpyxl의 셀 텍스트 추출로는 읽을 수 없지만, OCR을 돌리면 코드로 완전히 자동화할 수 있다. 2016년 파일에서 텍스트로 누락됐던 6개 전부를 `tesseract` + `kor` 언어팩으로 검증한 결과:

| 시도 | 확인된 행 | OCR 원본 결과 | 17개 후보 매칭 |
|---|---|---|---|
| 서울특별시 | ~61 | `서울특별시` | 일치 |
| 대구광역시 | 1,115 | (육안 확인, OCR 재검증 생략) | 일치 |
| 광주광역시 | 1,660 | `광주광역시` | 일치 |
| 울산광역시 | 2,686 | `올산광역시` (한 글자 오독) | 편집거리 최소값으로 자동 보정 → **울산광역시** |
| 세종특별자치시 | 2,994 | `세종특별자치시` | 일치 |
| 충청남도 | 4,594 | `충청남도` | 일치 |

6개 중 5개는 OCR이 정확히 맞혔고, 1개(울산)는 한 글자를 오독했다. 다만 **정답이 17개 시도명으로 정해진 폐쇄형(closed-set) 문제**이므로, OCR 결과와 17개 후보 사이 편집거리(fuzzy match)를 계산해 가장 가까운 후보로 자동 보정하면 사실상 100% 정확도를 얻을 수 있다. 전부 표준 행정구역 순서와도 일치했다.

> 참고: 이미지 안의 회색 반복 이미지 대부분(2016년 314개 중 다수)은 실제로는 **"Bridge Plan 2020" 워터마크와 "2016년 지방자치단체 시행계획" 페이지 헤더**였다. PDF→엑셀 변환 과정에서 매 페이지마다 삽입된 장식 이미지로, 항상 1번째 열(빈 칸)에만 붙어 있어 데이터 텍스트와 안 겹치고 텍스트 읽기에 영향이 없다. 진짜 시도명 타이틀 배너는 이 워터마크와 크기가 달라 구분 가능했다.

**OCR 자동화 파이프라인 (코드로 완결, Claude Code 불필요):**

1. 파일별 임베디드 이미지의 바이트 크기를 세어 최빈 크기(워터마크)를 제외
2. 남은 후보 중 각 시도 섹션 시작 지점 근처 이미지를 추출
3. `pytesseract`로 OCR 실행
4. OCR 결과를 17개 시도명 후보와 편집거리로 비교해 가장 가까운 후보로 확정 (`rapidfuzz` 사용)
5. 편집거리가 임계값 이상으로 먼 경우에만 사람 검수 플래그

**필요 환경**: 시스템에 `tesseract`(Homebrew: `brew install tesseract tesseract-lang`, Linux: `apt install tesseract-ocr tesseract-ocr-kor`) + Python `pytesseract`, `rapidfuzz` 패키지 (프로젝트 의존성에 추가 완료).

#### 3-2-2. 헤더/컬럼 구조가 연도마다 다름

> **`skiprows`란**: `pandas.read_excel()`이 표를 읽을 때 맨 위에서 몇 줄을 건너뛸지 지정하는 인자다. 원본 파일들은 실제 헤더 위에 "붙임 (OO시 OO교육청)", "(단위: 백만원)" 같은 타이틀/부제목 줄이 붙어 있는데, 이 줄 개수가 연도마다 달라서 `skiprows` 값도 파일마다 다르다.

| 연도 | skiprows | 헤더 행수 | 특이사항 |
|---|---|---|---|
| 2016~2019 | 1 | 2행 | `재정 구분` 컬럼에 `계/국비/지방비`가 있어 **세부사업 1개당 3행**으로 분리됨. `공통/자체` 컬럼 없음 |
| 2020 | 2 | 2행 | 참조 함수와 거의 일치 |
| 2021 | 3 | 2행 | 타이틀 위에 부제목 줄이 하나 더 있어 skiprows가 밀림 |
| 2022 | 1 | 2행 | |
| 2023 | 3 | 2행 | 컬럼명 자체에 시도명이 붙는 경우 있음(예: `세부사업명 (서울)`) → 컬럼명 정규화 필요 |
| 2024 | 2 | 2행 | 참조 함수 가정과 정확히 일치 |

`skiprows`는 파일마다 다르므로 고정값을 쓸 수 없고, 파일별로 자동 탐지하는 로직이 필요하다.

#### 3-2-3. (a)/(b) 연도 라벨 자동판별

`find_budget_columns` 로직(컬럼명에서 4자리 연도를 뽑아 큰 값=당해년도, 작은 값=전년도로 판별)은 그대로 유효함을 확인했다. 2016~2019년 파일은 `(A)/(B)` 라벨, 2022~2024년은 `(a)/(b)` 라벨을 쓰지만 둘 다 이 로직으로 커버된다.

#### 3-2-4. 중분류 번호는 라벨일 뿐, 대분류 간 중복될 수 있음 (데이터수집팀 교차검증)

데이터수집팀(최연호)이 확인해준 내용: 같은 중분류 번호(예: `1. 함께 일하고 돌보는 사회`)가 **대분류가 다르면 각각 별도로 존재**한다 — 공통사업 쪽에도 `1. 함께 일하고 돌보는 사회`가 있고, 자체사업 쪽에도 같은 번호가 따로 있다.

이는 처음부터 설계한 방침과 일치한다: `중분류` 값은 번호를 파싱해서 비교하는 게 아니라 **문자열 전체를 라벨로 취급**해 행 순서대로 ffill로 채우고, QA 대조도 `groupby(["대분류", "중분류"])`로 **대분류까지 같이 묶어서** 집계한다. 그래서 대분류가 다른 두 "1. 함께 일하고 돌보는 사회"가 서로 섞여 합산되는 일은 없다.

같은 확인 과정에서 **제3차(2016~2019)와 제4차(2020~2024) 기본계획 시기마다 표를 보여주는 방식이 달랐다**는 것도 교차검증됐다 (§3-2-2의 `계/국비/지방비` 구조 차이와 일치).

## 4. 처리 파이프라인

### 4-1. 범위 (Scope)

"1개 연도, 그 연도의 여러 시도" 시나리오는 **어느 연도를 골라도 실제로 가능**함을 확인했다 (파일 하나에 이미 17개 시도가 다 들어있음). 검증용 샘플은 아래로 진행할 것을 제안한다.

1. **2024년 · 서울/부산** — `정리본_자동` 경로로 우선 검증 (서울은 원본 파싱으로 이미 완료, 부산으로 `정리본_자동` 경로 자체를 검증)
2. **2019년 · 서울** — `정리본_자동`이 있어도 신뢰도가 낮을 수 있는 이미지 전용 시도명 케이스와 비교 대조
3. (2차 범위) 2016~2019년의 `계/국비/지방비` 3행 구조 — `정리본_자동`이 이 구조까지 흡수했는지 확인 필요

### 4-2. 실행 순서 — 권장 경로 (`정리본_자동` 활용)

1. **`정리본_자동` 시트 로드** — `pd.read_excel(path, sheet_name="정리본_자동", header=0)`. skiprows 자동탐지·헤더 평탄화 불필요
2. **시도별 필터링** — `지역` 컬럼으로 바로 분리 (경계 탐지·OCR 불필요)
3. **계층 태깅** — `classify_row`로 대분류_소계/중분류_소계/헤더반복/세부사업 판별 후 대분류/중분류 ffill (§4-4 기존 함수 재사용)
4. **`행유형` 교차검증** — "연속행 후보" 태깅된 행을 직접 확인해 진짜 페이지 잘림인지 VBA 오탐인지 판별 (§3-1 참고 — 맹신 금지)
5. **leaf/long 변환**
6. **QA 대조** — leaf 합계 vs 소계 행 값 검증
7. **원본(`Table 1`) 대조** — 같은 시도·연도를 §4-3 경로로도 한 번 돌려서 결과가 일치하는지 표본 검증 (최소 1개 시도는 두 경로 다 확인)
8. **저장** — `data/interim/{시도명}/`에 wide·long·필터링전 전체원본, `reports/`에 QA 검증결과

### 4-3. 실행 순서 — 대안 경로 (원본 `Table 1` 직접 파싱)

`정리본_자동`이 없는 연도가 나오거나, 위 7번 교차검증에서 불일치가 발견될 경우 사용한다. 2024년 서울 샘플에서 이미 검증 완료된 경로다.

1. **skiprows 자동탐지** — 파일별 헤더 시작 행 확인
2. **시도 경계 찾기** — 텍스트 정규식 우선, 텍스트 없는 구간만 OCR + 17개 후보 편집거리 매칭으로 보완 (§3-2-1)
3. **시도별 슬라이싱** — 확정된 경계 기준으로 시트를 시도별 원본 조각(raw slice)으로 분리
4. **조각별 정제** — 헤더 평탄화 → 계층 태깅 → 대분류/중분류 ffill → leaf/long 변환 (§4-4 참조 함수 재사용)
5. **QA 대조**
6. **저장**

### 4-4. 함수 재사용/신규 방침

| 함수 | 용도 | 비고 |
|---|---|---|
| `load_정리본(filepath)` | `정리본_자동` 시트 로드 + 컬럼명 표준화 | **신규**. 권장 경로 진입점. 시트가 없으면 예외 발생시켜 대안 경로로 넘어가야 함을 알림 |
| `verify_행유형_candidates(df)` | "연속행 후보" 행을 뽑아 실제 페이지 잘림인지 판별 지원 | **신규**. 완전 자동화보다는 사람 검수를 돕는 리포트 생성 목적 |
| `classify_row(세부사업명)` | 대분류_소계/중분류_소계/헤더반복/세부사업 판별 | **기존 재사용** — 2024 서울에서 검증 완료, 어느 경로든 그대로 사용 |
| `find_budget_columns(columns)` | 컬럼명에서 당해/전년도 예산 컬럼 자동판별 | **기존 재사용** — 두 경로 다 필요할 수 있음 |
| `detect_header_skiprows(filepath)` | 파일별 타이틀 행 개수 자동 탐지 | **신규**, 대안 경로(§4-3)에서만 필요 |
| `find_sido_boundaries(filepath)` | 시도 섹션 시작 행 탐지 (텍스트 정규식 + OCR 폴백) | **신규**, 대안 경로에서만 필요. 로직은 §3-2-1에 정리됨 |
| `flatten_columns`, `parse_budget_sheet`, `to_leaf`, `to_long`, `qa_check` | 헤더 평탄화·leaf/long 변환·QA | **기존 재사용**, 대안 경로 중심. `qa_check`(중분류 소계 매칭)는 두 경로 모두 시도 단위로 나눠서 실행 |

## 5. 산출물 (Deliverables)

이 저장소는 `README.md`에 정의된 `data/raw/` → `data/interim/` → `data/processed/` 3단 폴더 구조를 따른다. 이번 작업도 이 규칙을 지켜야 하며, 임의로 새 폴더를 만들거나 `data/raw/`에 있는 원본을 직접 수정/덮어쓰기 하지 않는다.

- **`data/raw/`**: 원본 엑셀 9개, **읽기 전용**으로만 사용 (git 추적 제외 대상이므로 이 안에서 아무것도 새로 만들지 않음)
- **`data/interim/`**: 시도별 결과는 `data/interim/{시도명}/` 하위 폴더에 모은다 (폴더만 봐도 어떤 시도인지 바로 알 수 있어 관리가 쉽다). 폴더로 시도가 구분되어도 파일명에 시도명을 한 번 더 넣어, 파일이 폴더 밖으로 나가거나 검색될 때도 바로 알아볼 수 있게 한다.
  - 시도 경계 탐지 결과 요약표(대안 경로 전용, 연도 전체 17개 시도를 한 번에 다룸): `data/interim/{연도}_시도경계_탐지로그.csv`
  - 시도별 wide(leaf) 정제 테이블: `data/interim/{시도명}/{연도}_{시도명}_세부사업_정제.csv`
  - 시도별 long(tidy) 테이블: `data/interim/{시도명}/{연도}_{시도명}_세부사업_정제_long.csv`
  - 필터링 전 시도별 전체 원본(소계·헤더반복 노이즈까지 다 포함, QA용 보관): `data/interim/{시도명}/{연도}_{시도명}_필터링전_전체원본.csv`
- **`data/processed/`**: 지금 당장은 사용하지 않음. 17개 시도 × 여러 연도 처리가 끝난 뒤, 전부 이어붙인 **하나의 최종 통합 테이블**을 만들 때 이 폴더에 저장한다. 시도별 개별 결과물은 그 통합 테이블을 만들기 위한 중간 조각이라 `data/interim/`에 둔다.
- **`reports/`**: 팀 공유용 QA 결과와 이 계획 문서 같은 리포트류
  - QA 대조 결과: `reports/{연도}_{시도}_QA_검증결과.csv` (검증 대상 전체 + 일치/불일치 결과 컬럼을 항상 저장 — 불일치가 0건이어도 "몇 건을 검증해서 다 맞았는지"가 파일 자체에 남도록, 불일치 행만 걸러서 저장하지 않음)

위 산출물을 §4-1의 1·2번 케이스(2024 서울/부산, 2019 서울) 분량으로 우선 생성해 팀 공유. 코드는 CLAUDE.md의 역할 분리 규칙에 따라 `src/features/`(파싱·경계탐지·피처), `src/evaluation/`(QA 대조)로 나눠 작성한다.
