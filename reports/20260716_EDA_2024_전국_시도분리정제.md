## 1. 이 작업이 왜 필요한가

#3에서 2024년 서울 1개 시도로 검증한 분리·정제 파이프라인을, 같은 2024년도 데이터의 전체 17개 시도로 확장하는 작업이다. 서울만 봤을 때는 안 보이던 문제(다른 시도의 예외 케이스, 원본 문서 자체의 결함)가 있는지 확인하는 것이 목적이다.

이번에는 원본 `Table 1`을 직접 파싱하는 대신, 데이터수집팀이 미리 `지역` 컬럼을 채워둔 `정리본_자동` 시트(`data/raw/칼럼정렬/`)를 표준 입력으로 사용했다 — 시도경계탐지·헤더평탄화 과정 없이 바로 지역별 분리부터 시작할 수 있기 때문이다.

## 2. 어떻게 처리했나

1. `정리본_자동` 시트 로드 (7,340행, 17개 시도, 12컬럼)
2. `지역` 컬럼 기준으로 17개 시도 분리
3. #3의 `classify_row`를 그대로 재사용해 대분류_소계/중분류_소계/헤더반복/세부사업 태깅
4. 대분류/중분류 라벨을 세부사업 행에 전파(ffill)
5. leaf(세부사업)만 추출
6. QA 대조: 중분류_소계 행 예산 vs leaf 합계

## 3. 서울에서는 안 보였던 문제와 해결

| 발견한 문제 | 원인 | 해결 방법 |
|---|---|---|
| "붙임 (OO시 OO교육청)" 시도경계 제목행이 `세부사업`으로 잘못 분류됨 | `정리본_자동`이 시도경계 텍스트를 완전히 제거하지 못하고 일부 남겨둠(17개 시도 중 16개, 서울 제외) | `classify_row`에 `sido_title_pattern`(#3에서 만든 시도명 정규식) 체크 추가, `헤더반복`으로 분류 |
| 경기·경남에서 `중분류_소계`가 8건이 아닌 9건 | "65.세 이상 건강증진 지원", "2.-2.경로당 환경개선 지원"처럼 세부사업명이 우연히 "숫자+마침표"로 시작해 중분류 헤더로 오분류됨 | 진짜 중분류 헤더는 전부 `(공통)`/`(자체)`로 끝난다는 공통점을 이용해 조건 추가 |
| `2024년 예산` 컬럼이 숫자와 문자열이 섞인 `object` 타입 | 예산이 0원 또는 해당없음인 경우 "-"로 표기됨 | 콤마 제거 후 `pd.to_numeric` 변환, "-"는 0으로 처리 |

## 4. QA 결과

**검증 대상(중분류 단위) 136건 중 114건 일치, 22건 불일치.**

불일치는 성격이 다른 3가지로 나뉜다.

### 4-1. 미세한 반올림 오차 (13건)

부산·경기·충북·전북·전남 등에서 ±1~6원 수준의 차이. 금액 단위가 백만원인 걸 감안하면 원본 문서 자체의 반올림 오차로 보인다 — 여러 시도에 흩어져 있고 규모가 작아 파이프라인 버그로 보기 어렵다.

### 4-2. 정상 케이스 (1건)

광주 "4.인구구조 변화에 대한 적응(공통)"은 원본 소계가 0원이고 대응하는 leaf 항목이 없다. `groupby`가 그 조합을 만들지 않아 `NaN`으로 나타난 것뿐, 실제 문제는 아니다.

### 4-3. 충남 — 원본 문서 자체의 결함 (8건 전부)

충남은 8개 중분류 전부에서 leaf 합계가 소계보다 크게 부족했다(최대 -596,452). `classify_row` 분류 개수(세부사업 106 + 중분류_소계 8 + 대분류_소계 2 + 헤더반복 1 = 117, 원본 행수와 일치)와 숫자 변환 모두 정상이라, 원본 `Table 1`을 직접 대조했다.

**"3. 모두의 역량이 고루 발휘되는 사회(공통)" 소계 행(19,928원) 전후 (Table1 엑셀행 5355~5361)**

| 엑셀행 | 세부사업명 | 공통/자체 | 예산 |
|---|---|---|---|
| 5354 | 희망소리 찾기 사업(보청기 지원사업) | 공통 | 8 |
| 5355 | 세부사업명 | 사업 분류 | 2024년 예산 |
| 5356 | (병합셀) | 공통/ 자체 | 총예산(a) |
| **5357** | **3. 모두의 역량이 고루 발휘되는 사회(공통)** | - | **19928** |
| **5358** | **4.인구구조 변화에 대한 적응(공통)** | - | 17795 |
| 5359 | 한부모가족자녀의 안정적 성장 지원 | 공통 | 764 |
| 5360 | 한부모가족 생활안정 지원 | 공통 | 495 |

소계 행(5357) 바로 다음이 곧바로 다음 중분류의 소계 행(5358)이다 — **원본 문서 자체에 이 중분류에 속하는 개별 세부사업이 단 한 건도 기재돼 있지 않다.** 소계 금액만 있고 이를 구성하는 세부사업 목록 자체가 원본에 없는 것이다.

참고로 원본행 번호가 중간에 비어 보였던 두 구간(5312~5315, 5333~5336)은 실제로는 데이터 손실이 아니라 페이지가 바뀌며 반복된 헤더 줄("세부사업명 / 공통·자체 / 2024년 예산")이었다 — `정리본_자동`이 정상적으로 걸러낸 것.

**결론**: 충남의 QA 불일치는 `classify_row`/`assign_labels`/`정리본_자동` 변환 과정의 버그가 아니라, **원본 시행계획 문서 자체가 일부 소계와 세부사업 목록을 안 맞게 기재한 경우가 있다는 것**을 뜻한다. 코드로 고칠 대상이 아니다.

## 5. 증감액·증감율 컬럼 부호 소실 (전체 시도 확인)

같은 충남 "3. 모두의 역량이 고루 발휘되는 사회(공통)" 행에서, 팀원이 증감액·증감율 부호 문제를 지적했다(2024년 예산 19,928 < 2023년 예산 42,867.5이면 감소인데 증감액·증감율이 양수로 표기됨). 전체 17개 시도로 확인 범위를 넓혔다.

**방법**: 세부사업별로 `당해년도 예산 - 전년도 예산`을 직접 계산해, 결과가 음수(예산 감소)인 행만 골라서 원본 `증감액` 컬럼이 실제로 음수로 찍혀 있는지 대조.

**결과 — 시도별 부호 소실 비율(예산이 감소했는데 증감액이 양수로 찍힌 비율)**

| 지역 | 부호 소실 비율 |
|---|---|
| 인천·경남·충남·대구·대전·부산·전북 | 100.0% |
| 전남 | 83.3% |
| 제주 | 64.2% |
| 서울 | 61.6% |
| 광주 | 7.9% |
| 충북 | 1.3% |
| 울산·세종·경기·경북·강원 | 0.0% |

일부 시도(강원 등)는 `증감액`에 마이너스 부호가 정상적으로 붙어 있는 반면(예: `-981`), 다른 시도(부산 등)는 계산상 명백히 감소인데도 `증감액`에 양수(`3765`)만 적혀 있다. 0%와 100%로 극명하게 갈리는 패턴을 보면 무작위 오류가 아니라 **시도별로 원본 문서에서 감소분을 표기하는 방식 자체가 달랐을 가능성**이 높다(예: 일부 시도가 색상 등 마이너스 부호 이외의 방식으로 감소를 표시했고, PDF→엑셀 변환 과정에서 그 정보가 사라졌을 가능성).

추가로, 부호만 다른 게 아니라 **절댓값 자체가 다른 경우도 75건(전체의 약 1%)** 확인됐다(예: 대구 "아이돌봄 지원사업(구군)"은 계산상 증감액이 +7,614여야 하는데 원본엔 `79`로 적혀있음 — 부호 문제가 아니라 별개의 데이터 오류). 이 때문에 "부호만 붙이는" 방식 대신 **전 항목을 재계산하는 방식**을 택했다 — 절댓값이 맞는 나머지 99%는 재계산 값과 원본 값이 같으니 손해가 없고, 절댓값까지 틀린 1%도 같은 규칙 하나로 해결된다.

**결론 및 권장 조치**: 원본의 `증감액`/`비율` 컬럼은 신뢰할 수 없다. 이 컬럼을 그대로 쓰지 말고, **`당해년도 예산 - 전년도 예산`을 직접 재계산해서 사용**해야 한다 — 이렇게 하면 시도별 표기 방식 차이와 무관하게 항상 정확한 방향(증가/감소)이 나온다.

## 8. 세부사업명 오탈자·중복 후보 탐지

세부사업명 오탈자는 자동으로 고치지 않고, 사람이 검토할 후보만 찾는 방식으로 접근했다(대분류/중분류에도 동일하게 줄바꿈 정규화를 적용해두어, 이 비교의 전제인 `clean_text` 정제 범위를 세부사업명·주요내용·대분류·중분류로 넓혔다).

**방법**: 같은 지역·중분류 안에서 `rapidfuzz.fuzz.ratio`로 완전히 같지는 않지만 유사도가 높은(90점 이상) 세부사업명 쌍을 찾는다. 공백·문장부호(쉼표, 가운뎃점, 마침표, 슬래시, 하이픈, 괄호)만 다른 경우는 정규화 후 비교해 검수 대상에서 제외한다 — 단순 표기 차이는 사람이 볼 필요가 없기 때문이다.

**결과**: 초기 316건 → 문장부호 정규화 반영 후 224건으로 감소.

**팀원(최연호) 피드백**: 이름만으로 판단하지 말고 `세부사업명 - 당해예산 - 주요내용` 조합을 함께 봐야 정확하다는 의견 — 이름이 비슷해도 예산·주요내용이 다르면 별개 사업일 수 있다. 이 의견을 반영해 후보 목록에 `당해예산`, `주요내용`을 함께 출력하도록 확장했다.

**신뢰도 필터**: 이름은 다르지만 `당해예산`까지 완전히 같은 쌍을 우선순위로 본다 — 다만 예산이 0원(원본에 "-"로 표기된 "해당없음" 케이스, `to_numeric_budget`이 0으로 변환)인 경우는 우연히 일치할 확률이 높아 신뢰도가 낮다. 0이 아닌 금액까지 일치하는 경우만 신뢰도 높은 후보로 취급한다.

## 9. 주요내용 원자화(지원대상/지원내용 분리) 시도 및 보류

DB화를 염두에 두면 `주요내용`("지원대상 : X 지원내용:Y" 형식)의 두 항목을 별도 컬럼으로 분리(원자화)하는 게 정규화 원칙에 맞다고 보고, 분리 전에 이 패턴이 실제로 얼마나 지켜지는지 먼저 확인했다.

**방법**: `^지원대상\s*[:：]\s*(.*?)\s*지원내용\s*[:：]\s*(.*)$` 정규식으로 전체 leaf 데이터의 `주요내용`을 검사.

**결과**: 일치 18.7%, 불일치 79.4%, 결측 1.9% — 예상보다 훨씬 낮은 커버리지.

**원인(불일치 샘플 확인 결과)** — 정리하면 표기 형식이 최소 4가지로 갈린다:
1. **라벨 단어 자체가 다름**: 대부분은 "지원대상/지원내용"이지만, 일부는 "**사업대상 : ... 사업내용:...**"처럼 다른 단어를 씀
2. **라벨이 아예 없는 행이 상당수**: "공공실버케어센터 확충", "교육공무원의 육아기 근로시간 단축제도 운영"처럼 라벨 구조 없이 짧은 설명 문장만 있는 행이 많음
3. **라벨이 하나만 있고 나머지는 생략**: 예) "지원대상 : 만 65세 이상 국민기초생활수급자, 차상위계층, 기초연금 수급자 중 돌봄이 필요한 어르신 대상자의욕구를반영하여각분야별서비스동시제공" — "지원대상"만 있고 "지원내용" 라벨 없이 내용이 바로 이어붙음
4. 라벨이 있어도 콜론 없이 줄바꿈으로만 구분되는 등 표기 형식이 제각각

**라벨 변형까지 포함한 재검증**: "지원대상|사업대상 ... 지원내용|사업내용" 로 범위를 넓혀도 일치 18.7% → 20.9%로 소폭 상승에 그침(불일치 79.4% → 77.2%). 라벨 표기 문제가 원인의 전부가 아니라, **원본 데이터 자체에 이 구조를 따르지 않는 행이 다수**라는 게 재확인됨.

**결론**: "지원대상 : X 지원내용:Y" 구조는 전체 데이터의 일부(약 20%)에서만 성립하는 패턴이지 보편적 구조가 아니다. **전면 원자화·형식통일은 보류**하고, `주요내용`은 비정형 자유텍스트로 취급해 Level2 분류의 보조 입력으로만 사용하는 현재 방식을 유지하기로 함.

## 9-1. 지원대상/지원내용 정규식 확장 및 부분 활용

전면 원자화는 보류했지만, 패턴이 맞는 행에 한해서는 구조화된 값을 뽑아두는 게 이후 분석에 도움이 된다고 보고 두 가지를 추가했다.

- **라벨 중복 제거**: "지원대상 : 지원대상 : ..."처럼 같은 라벨이 연속으로 중복된 경우를 `dedup_label`로 먼저 정리해, 패턴 매칭이 라벨 중복 때문에 실패하는 경우를 없앴다.
- **라벨 범위 확장**: 행 1370("대상 : 0세 이하 영유아 내용:...")처럼 "지원대상/사업대상"이 아니라 짧게 "대상"/"내용"만 쓰는 표기가 있는 걸 확인해, `support_pattern_broad`에 세 번째 대안으로 추가했다 — `^(지원대상|사업대상|대상)\s*[:：]?\s*(.*?)\s*(지원내용|사업내용|내용)\s*[:：]?\s*(.*)$`.

이 정규식에 걸리는 행만 `extract_via_regex`로 `지원대상`/`지원내용_상세` 컬럼을 채우고, 안 걸리는 행은 강제로 분리하지 않고 `NaN`으로 남긴다 — 나머지 80%를 억지로 쪼개서 잘못된 값을 만드는 것보다, 확실한 20%만 구조화하고 나머지는 원문 그대로 두는 쪽이 안전하다고 판단했다.

## 9-2. 주요내용 LLM 기반 오탈자·공백 정제 (업스테이지 Solar Pro 3)

**배경**: `주요내용` 원문에 OCR 변환 과정에서 생긴 것으로 보이는 오탈자(단어 내부에 잘못 들어간 공백, 줄바꿈으로 끊긴 단어)가 다수 확인됐다. 표기 형식이 워낙 다양해(9-1 참고) 정규식만으로는 이런 오탈자를 일괄 교정할 수 없어, 자연어를 이해하는 LLM으로 텍스트만 교정하는 방법을 검토했다.

**데이터 안전성 원칙 예외 처리**: CLAUDE.md `01_data_safety` 규칙상 원칙적으로 외부 API로 데이터를 보내는 코드는 작성하지 않는다. 다만 `주요내용`은 지자체 예산 사업 설명으로 PII가 없는 공개 행정 데이터이고, 담당자(이정연)가 학생 신분으로 무료로 쓸 수 있는 업스테이지 Solar API를 이 용도에 한해 명시적으로 예외 승인했다. 이 예외는 **이 파이프라인의 `주요내용` 텍스트 정제 용도에 한정**되며, 다른 컬럼이나 다른 작업에 일반화된 승인이 아니다.

**아키텍처 — 역할을 명확히 분리**:

| 역할 | 담당 | 이유 |
|---|---|---|
| `지원대상`/`지원내용_상세` 분리 | 정규식(`extract_via_regex`, 9-1) | 이미 검증된 결정론적 로직이 있고, 패턴이 없는 행에 LLM을 쓰면 과잉분리(원문에 없는 경계를 만들어내는) 위험만 커짐 |
| `주요내용_정제`(오탈자·공백 교정) | LLM (`solar-pro3`) | 정규식으로 표현하기 어려운 자연어 수준의 교정이 필요 |

- `response_format`(`json_schema`)으로 API 단에서 출력 구조를 강제한다 — `solar-pro3`/`solar-pro-2`만 지원. 이 옵션 전에는 모델이 마크다운 코드펜스로 감싸거나 JSON 뒤에 군더더기 설명을 붙이는 문제가 있었는데, 구조화 출력으로 해결했다.
- 프롬프트는 숫자·금액·비율·날짜·기관명·지역명·사업명·법령명을 절대 바꾸지 않도록 명시하고, 확신 없으면 원문을 그대로 반환하도록 지시한다. `temperature=0`으로 고정.
- 파싱 실패 시 1회 재시도, 그래도 실패하면 원문을 그대로 사용한다(정제 결과가 결측으로 남는 일이 없게).
- **자동 QA**: 정제 전후로 `re.findall(r"\d+", text)`를 비교해 숫자 시퀀스가 그대로 보존됐는지 자동 검증한다. 다르면 `숫자보존=False`로 표시해 검토 대상으로 남긴다 — LLM이 금액을 바꾸는 등의 실수를 자동으로 잡아내는 안전장치.
- 설정은 기존 `configs/` 관례에 맞춰 `configs/llm_extraction.yaml`로 외부화했다(모델·temperature·프롬프트·응답 스키마·샘플 파라미터).

**개발 과정에서 발견한 문제와 반영한 외부 리뷰**: 초기 프롬프트는 라벨("세부사업명:", "주요내용:")을 출력에 그대로 베끼거나, 라벨이 하나만 있는 문장을 억지로 두 개로 쪼개는 문제가 있었다. 외부 코드 리뷰를 받아 다음을 반영해 아키텍처를 다시 잡았다: (1) LLM이 지원대상/지원내용 라벨 분리를 아예 시도하지 않고 정규식 결과만 쓰도록 역할 자체를 분리, (2) JSON 강제를 프롬프트가 아닌 API 레벨(`response_format`)로, (3) 실패 시 폴백을 원문 유지로, (4) 숫자 보존 자동 검증 추가, (5) 프롬프트 필드명을 "원문"/"세부사업명(참고용)"으로 바꿔 라벨 복사 문제 해결.

**검증**: 20건 무작위 샘플(`random_state=42`)에 대해 20/20 성공, 숫자 불일치 0건 확인.

**전체 실행**: 7,154건 전체에 적용 시 약 70~90분 소요(초당 1.3~1.7건 처리 기준). 중간에 끊겨도 이어서 진행할 수 있도록 200건 단위로 `data/interim/llm_정제_체크포인트.csv`에 진행 상황을 저장하는 체크포인트 로직을 추가했다.

**전체 실행 결과 (2026-07-16 완료)**:

- 7,154/7,154행 처리 완료. `주요내용_정제` 결측 133건은 원본 `주요내용` 자체가 결측인 행과 정확히 일치(17개 시도 파일 대조 확인) — API 실패가 아니라 정상적인 "결측 유지" 케이스.
- `숫자보존` 불일치는 전체 중 1건(충북, "결혼이주여성 지원"): 원본 `1가구당3,00천원이내왕복항공료지원`을 LLM이 `1가구당 3,000천원 이내`로 교정해 숫자를 임의로 바꿈. 당해예산(37백만원) 대비 3,000천원×7가구=2,100만원(57%)이 300천원×7가구=210만원(5.7%)보다 그럴듯해 보여 처음엔 LLM 값이 맞다고 판단했으나, **담당자가 원본 문서를 직접 재확인한 결과 `3,00천원`이 실제 원문 표기임을 확인**했다. 최종 CSV의 `주요내용_정제`는 원본 숫자 그대로(`3,00천원`, 띄어쓰기만 정리) 되돌려 반영했다.
  - **교훈**: 숫자 불일치는 예산 규모 등 정황 추정만으로 판단하지 말고 반드시 원본 문서 대조로 확정해야 한다. 이번 사례는 정황상 LLM 값이 맞아 보였지만 실제로는 틀렸다.
- 17개 시도 CSV(`data/interim/{시도명}/2024_{시도명}_세부사업_정제.csv`)에 `주요내용_정제`/`지원대상`/`지원내용_상세` 컬럼을 포함해 재저장 완료(§6 최종 스키마 참고). 합계 7,154행으로 원본과 일치.
