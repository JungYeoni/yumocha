# 2020년도 전국 시도 세부사업 예산현황 분리·정제

## 1. 입력과 연도 설정

- 입력 파일: `세부사업표추출_2020년도 지방자치단체 저출산고령사회 시행계획 (제3차 기본계획)_칼럼정렬.xlsx`
- 입력 시트: `정리본_자동`
- 입력 규모: 7,003행, 12개 컬럼, 17개 시도
- 예산 컬럼: `2020년 예산`, `2019년 예산`
- 0원 토큰: `-`
- 단위표기 머리글: `UNIT_NOTATION_PATTERN` 적용
- 재원구분 정리: `NEEDS_FUNDING_SOURCE_CLEANUP=False`

`사업분류재정구분`의 비결측값은 `공통`, `자체`뿐이다. 계·국비·지방비 구조가 없으므로 재원구분 합산 유틸은 적용하지 않았다.

## 2. 2020년 행 유형 보정

공통 함수는 `(공통)/(자체)`로 끝나는 중분류를 인식하지만 2020년 원본은 `(공통사업)/(자체사업)` 형식이므로 노트북에서만 보정했다. 공통 모듈은 수정하지 않았다.

최종 행 유형은 다음과 같다.

| 행 유형 | 행 수 |
|---|---:|
| 세부사업 | 6,858 |
| 중분류 소계 | 100 |
| 대분류 소계 | 34 |
| 머리글 반복·제외 | 11 |

제외 11건은 서울 단위표기 1건, 경기 도·시/도비·시비 보조 소계 6건, 인천·세종·경기 전체 총계 3건, 경남 원본행 7,494의 잘못된 중분류 머리글 1건이다.

경남 원본행 7,494는 `3. 인구구조 변화 적극 대비 (자체사업)`로 적혀 있지만 뒤에 노후사업이 이어지고 실제 3번 소계가 원본행 7,651에 다시 나온다. 반복된 원본 오기 머리글로 제외했다.

- 절대 오차율 10% 초과: 3개

세 건은 정제 결과를 자동 보정하지 않고 원본 불일치로 유지했다.

## 4. 텍스트 정제와 LLM 상태

- 세부사업명·주요내용 공백 및 선행 불릿 정리 완료
- 정규식 기반 지원대상·지원내용 분리 완료
- 유사 사업명 검토 후보: 250건
- 실제 외부 API 호출: 0건
- `RUN_LLM=False` 실행에서는 `주요내용_정제`에 원문을 유지

LLM 체크포인트 로직은 가짜 호출 함수로 검증했다. 체크포인트는 인덱스만 믿지 않고 `지역`, `원본행`, `세부사업명`을 함께 대조한다.

- 첫 실행: 체크포인트 6,858행 생성
- 실제 텍스트 호출 대상: 6,849건, 주요내용 결측 9건은 호출 생략
- 두 번째 실행: 체크포인트 6,858행 재사용, 신규 호출 0건
- 숫자 보존: 6,858건 통과
- 테스트 체크포인트: 검증 후 삭제

## 5. 비-LLM 산출물 검증

- wide: 17개 파일, 6,858행, 공통 15개 컬럼
- long: 17개 파일, 13,716행, 15개 컬럼
- 필터링 전 전체 원본: 17개 파일, 7,003행
- QA·행 유형 검토표 포함 총 53개 CSV가 UTF-8-SIG
- 같은 노트북을 다시 실행한 뒤 53개 CSV의 SHA-256 변화 0건

코드 검증 결과:

```text
ruff check: 통과
pytest: 82 passed
비-LLM 전체 셀 순차 실행: 통과
```

현재 실행 환경은 Jupyter 커널 통신 소켓이 차단되어 `nbconvert --execute`가 커널 시작 단계에서 중단됐다. 동일 코드의 독립 네임스페이스 순차 실행과 산출물 재실행 동일성은 통과했다.

## 6. 남은 작업

1. 로컬 `.env`의 Upstage API 키를 확인하고 `RUN_LLM=True`로 실제 체크포인트 실행
2. LLM 숫자 불일치가 발생하면 원문 복구 및 검토표 확인
3. 실제 LLM 결과로 wide·long 파일 재생성
4. 커널 재시작 후 전체 실행 및 최종 산출물 수치 재확인
5. 승인된 파일만 커밋·푸시하고 PR 생성
